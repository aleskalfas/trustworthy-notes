# ADR-005: Send Feedback desktop launcher and OS-integration trust posture

- Status: Accepted
- Date: 2026-06-20 (Windows shortcut), 2026-06-21 (macOS droplet)

**In one minute:** trustworthy-notes puts a "Send Feedback" entry on the user's desktop — the first time it writes an OS-integration artifact onto the machine outside its own config and workspace. This extends ADR-001's distribution posture (an unsigned, self-upgrading exe trusting GitHub-over-TLS) to a new trust surface. The launcher takes a different shape on each OS but the same posture: on Windows, first-run onboarding offers a `.lnk` shortcut created by handing a string to PowerShell — an OS program present on every Windows 10/11 — rather than bundling a COM library; on macOS, the install/bootstrap step compiles an AppleScript droplet with `osacompile`, the OS's own always-present tooling. Neither path adds a dependency to the PyInstaller build (preserving ADR-002). Both are user-initiated, never silent. The Windows shortcut targets the stable `tnotes.exe feedback` so it survives the upgrade rename; the macOS droplet bakes the absolute path of the installed `tnotes`, resolved in the user's shell at install time.

## Context

`tnotes feedback` is the second-most-prominent thing a non-technical user does with the tool (after running the pipeline), but nothing surfaces it: there is no installer, no Start-menu entry, no account. To make feedback discoverable, first-run onboarding offers to drop a "Send Feedback" shortcut on the user's desktop.

This is a new kind of action for the tool. Until now, everything trustworthy-notes writes lands in its own config or workspace; a desktop `.lnk` is the first artifact it places into the user's OS shell — an **OS-integration artifact** outside its own footprint. That extends the trust posture ADR-001 already stated for distribution: an **unsigned**, self-replacing exe whose only trust root is GitHub plus TLS. Writing into the user's desktop is a smaller surface than self-rewriting over the network, but it is a surface, and creating it deserves the same deliberate posture rather than a silent side effect.

The forces in tension:

- **Discoverability of feedback for one non-technical user** versus adding any new machinery — a dependency, a silent OS write — to a tool whose whole shape is "one unsigned exe, no installer."
- **A shortcut is inherently a Windows-only artifact**, so some Windows-specificity is unavoidable; the question is where it lives and how tightly it is confined.
- **The shortcut must keep working across upgrades**, where ADR-001's swap renames the running exe (`tnotes.exe` → `.old` → new exe into the freed name). A shortcut that encodes the wrong target is orphaned by the very upgrade mechanism ADR-001 built.

**The same need exists on macOS, where the author develops the tool.** A "Send Feedback" desktop launcher should exist there too, but the artifact, the place it is created from, and the OS trust gates are all different, so the cross-OS posture has to be stated rather than assumed to carry over.

- **The launcher's creator differs by OS, and this is not interchangeable.** On Windows the `.lnk` is offered by the windowless onboarding flow — but that flow runs *only* on the Windows windowless launch and never executes on macOS. A droplet hung off onboarding would therefore be dead code on macOS. The macOS launcher must be created by the install/bootstrap step instead: `scripts/bootstrap.sh` calls a new `tnotes install-droplet` command after installing `tnotes`.
- **A macOS desktop launcher is inherently an `.app`-shaped artifact**, just as a shortcut is inherently a `.lnk` — so some macOS-specificity is unavoidable; again the question is where it lives and how tightly it is confined.
- **A Finder-launched `.app` does not inherit the user's shell PATH.** It gets the minimal launchd environment, so a launcher invoking a bare `tnotes` would not resolve it — the macOS analogue of the Windows "the target must be stable" problem, but rooted in environment rather than the upgrade rename.

## Decision

**A PowerShell shell-out, not a bundled COM library.** The shortcut is created by invoking `WScript.Shell.CreateShortcut` through `powershell -Command ...` as a subprocess — **not** by bundling a Windows COM Python library (`win32com` / `pywin32`). This keeps the Python side OS-neutral and adds **no new dependency to the PyInstaller bundle**, whose frozen-build correctness ADR-002 must otherwise keep intact. PowerShell ships on every Windows 10/11, so the dependency is the OS itself. Because a shortcut is inherently Windows-only, this confines the unavoidable Windows-specificity to a single string handed to an always-present OS program, **gated so it never runs off-Windows** — consistent with how `winlaunch.py` already gates its `GetConsoleProcessList` ctypes call behind a platform check.

**User-confirmed, never silent.** The shortcut is created only behind a **one-tap confirm during onboarding**. Nothing is written to the user's desktop without their agreement, matching the user-initiated principle ADR-001 applied to upgrades: the tool reports and offers, the human decides. A tool that silently plants OS artifacts is the behaviour we are deliberately not adopting.

**The target is the stable exe name plus subcommand, not a versioned path.** The shortcut points at `tnotes.exe feedback` — the stable executable name, not a versioned or absolute build path. This is what makes it survive ADR-001's upgrade swap: the upgrade renames the live exe out of the way and moves the new build into the **freed, stable name**, so a shortcut that targets that name keeps resolving across every upgrade. The failure mode this avoids is explicit: a shortcut encoding a versioned path that ADR-001's rename swap would orphan on the next upgrade.

**Windows onboarding now offers two shortcuts, same posture and mechanism (issue #105).** Alongside "Send Feedback", the combined one-tap confirm also drops a **"Make Notes"** shortcut whose `.lnk` targets the stable exe with **no** baked argument, so dropping a PDF on it launches the bare exe with the file (→ the run/orchestrator flow). Both are built by the same `winlaunch._create_shortcut` PowerShell primitive, differing only in the `.lnk` name and the baked `Arguments` (none for Make Notes, `feedback` for Send Feedback), and are attempted independently so one failing never skips the other — the Windows parity of the macOS two droplets above. Everything stated for the single Windows shortcut — the PowerShell-over-bundled-COM mechanism, the no-new-dependency posture, the user-confirmed-never-silent gate, the stable-target survivability — applies to both unchanged.

### macOS: the Send Feedback droplet

**An `osacompile`'d AppleScript droplet, not a bundled app framework.** On macOS the launcher is an AppleScript droplet — an `on open` handler that does `tell application "Terminal" to do script "<tnotes> feedback <dropped>"` — compiled to an `.app` with `osacompile`, the OS's own always-present scripting toolchain. This is the direct macOS analogue of the Windows "PowerShell over a bundled COM library" choice: we lean on tooling present on every machine rather than bundling a framework, so **no new dependency enters the PyInstaller build** and ADR-002's frozen-build correctness stays intact on the macOS side too.

**Created by bootstrap, not by onboarding.** The droplet is produced by `tnotes install-droplet`, which `scripts/bootstrap.sh` invokes after installing `tnotes`. This is the deliberate consequence of the creator-differs-by-OS force above: the onboarding flow that creates the Windows `.lnk` never runs on macOS, so binding the droplet to bootstrap is what keeps it from being dead code. Running `install-droplet` — explicitly, or via the bootstrap the user chose to run — is itself the user-initiated action, the same "user-confirmed, never silent" principle the Windows path applies, just sited at install time instead of first-run onboarding.

**The macOS bootstrap installs two launchers, same posture and mechanism (issue #102).** Alongside "Send Feedback", `install-droplet` also drops a **"Make Notes"** droplet that runs the bare `tnotes <pdf>` book flow on a dropped PDF. Both are `osacompile`'d AppleScript droplets built by the same primitive, differing only in the `.app` name and the baked subcommand (none for Make Notes, `feedback` for Send Feedback); both bake the absolute install-time `tnotes` path and are attempted independently so one failing never skips the other. Everything above — the trust posture, the no-new-dependency mechanism, the one-time Terminal-automation consent, the re-bake-on-move recovery — applies to both unchanged.

**The target is the absolute `tnotes` path resolved at install time, baked into the droplet.** Because a Finder-launched `.app` gets only the minimal launchd PATH and not the user's shell PATH, a bare `tnotes` would not resolve. So `install-droplet` resolves the absolute path of the installed `tnotes` *in the user's shell* at install time and bakes it into the droplet's script. This is the macOS analogue of the Windows "stable target" choice — same goal (a target that resolves when the launcher fires), different root cause (environment isolation rather than the upgrade rename).

## Consequences

- **The shortcut is a second launch entry point with its own trust posture.** Beyond the main pipeline launch, the user now has a desktop entry that maps directly to the `feedback` subcommand — and feedback carries a distinct trust posture from the pipeline (it can exfiltrate data, per ADR-003). The shortcut surfaces that path one tap from the desktop; it does not change what `feedback` does or the consent gate that stands in front of every upload.
- **A moved exe leaves a dangling shortcut.** If the user relocates `tnotes.exe`, the shortcut breaks. This is accepted: re-running onboarding (or re-creating the shortcut) is the recovery, and the alternative — chasing the exe's location — reintroduces exactly the brittle path-coupling the stable-name target avoids.
- **Shortcut creation is validated last, on real Windows.** macOS cannot validate `.lnk` creation at all — there is no shell to write to and no PowerShell semantics to exercise — so the code is authored with the Windows path gated and mocked, and the **first real validation is a Windows run of the frozen exe**. This mirrors `winlaunch.py`'s own honesty about its Windows-only console code and ADR-001's running-exe swap: a known validation gap carried deliberately, not an oversight.
- **Hand-writing the raw `.lnk` binary was considered and rejected.** Emitting the shortcut's binary format directly in pure Python would also add no dependency, but it reimplements an undocumented, fragile binary format that the OS already knows how to produce correctly. Delegating to `WScript.Shell` via PowerShell buys the same zero-dependency outcome while letting Windows own the format.
- **The macOS droplet has no Gatekeeper "unidentified developer" dance — a different trust story from the downloaded exe.** Gatekeeper quarantine applies to *downloaded* files; a droplet **created locally by bootstrap** is never quarantined, so there is no unidentified-developer prompt. This is distinct from ADR-001's Windows SmartScreen story, which exists precisely because the exe arrives over the network. The one OS gate that does fire is a **one-time Terminal-automation (TCC) consent** on the first drop — the macOS analogue of the user-confirmed gate, surfaced by the OS rather than by us.
- **A `tnotes` reinstalled elsewhere leaves a stale droplet.** Because the absolute path is baked at install time, moving or reinstalling `tnotes` to a new location orphans the droplet's target. The recovery is to re-run bootstrap, which re-resolves and re-bakes the path — the macOS mirror of the Windows dangling-shortcut recovery (re-running onboarding). The accepted tradeoff is the same: re-stamping on move beats chasing the binary's location.
- **Droplet creation is validated on macOS; shortcut creation on Windows; neither validates the other.** The author works on macOS, so the droplet path gets real validation there while the Windows `.lnk` path remains authored-and-mocked until a real Windows run (above). The honesty is symmetric: each OS launcher is first exercised for real only on its own platform, and the cross-OS posture in this ADR is what keeps the two paths coherent in the meantime.
- **macOS distribution is not solved here — only install-from-source.** ADR-001 stays **Windows-exe-only**; the droplet serves the macOS **install-from-source** path (bootstrap over a checkout), not a packaged macOS download. The absence of a macOS distribution artifact is a known, deferred gap recorded here, not closed by this ADR. A future decision to ship a macOS build would revisit both ADR-001's distribution posture and the droplet's quarantine assumption above.
