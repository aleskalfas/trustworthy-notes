## Context

[COR-019](COR-019-schema-reference-form.md) fixed the typed-token reference form `[<namespace>:<id>]` and a resolver that confirms each id exists in the namespace owner's id collection — reading that collection from the namespace-declaring schema's own YAML, in sibling-file scope. [COR-023](COR-023-schema-binds-inline.md) then added the *adopter-data* layer: an adopter data file binds to a capability schema (a `pkit_schema:` field, or a `binds_to:` path-glob fallback the schema declares), and `pkit data validate` checks that file's shape against the bound schema.

Combining the two produces a relationship neither validates: adopter data that carries cross-file references. A token in one bound file points at an entry in a *sibling bound file* — e.g. a `[transport:<id>]` token in a trip's aggregator file referencing an entry in that trip's transport file. `pkit data validate` checks shape only; COR-019's resolver walks capability `schemas/` directories, not adopter trees. So a dangling reference passes silently — the precise failure referential-integrity validation exists to catch.

Two facts shape the fix. First, a capability schema that *describes* adopter data keeps its own id collection **empty by design** — the entries live in the adopter tree (that is what `binds_to:` is for). So a reference must not resolve into the schema's own collection (always empty → always dangling) but into the bound data instance. Second, adopter data freely carries *other* bracketed tokens that are **not** references — in prose fields, free-text values, or comments — so a resolver that treated every `[x:y]`-shaped string as a reference would misread them.

## Decision

**In adopter data, a typed reference resolves *through the binding to the bound instance*, not into the namespace-declaring schema's own collection. Resolution runs only at fields the citing schema marks reference-bearing; it locates the namespace's id pool as the **union of files bound to that namespace's schema within the validation scope** (the subtree `pkit data validate` is run on), reading each via its `x-pkit-id-collection`, and checks the id against that pool.** `pkit data validate` performs this as a default-on pass (`--shape-only` skips it).

1. **Position-gate annotation.** A field that holds references is marked on the citing schema's companion with `x-pkit-reference-namespace: "<namespace>"` — a value-position annotation mirroring the existing key-position `x-pkit-keys-from-namespace` ([COR-019]). The resolver inspects **only** annotated fields. It is a *position-gate*, not a namespace-source: the namespace is already in the token; the annotation declares that this position holds references at all.

2. **Resolution target is the bound instance.** The token's namespace resolves to its schema (by stem, as the existing resolver does); the id is then sought in the adopter file *bound to that schema*, following its companion's `x-pkit-id-collection` — not in the schema's own (empty) YAML. This refines COR-019's resolution target for the adopter-data case; see Rationale.

3. **Resolution scope is the validated subtree.** A namespace's id pool is the **union of every file bound to that namespace under the directory `pkit data validate` is run on**. A token resolves iff its id is in that pool. Isolation is by *what you validate*: a narrower target is a stricter universe (validate one instance's subtree and only its data is in scope); a wider target deliberately unions more (validate a parent and a shared file under it serves every instance below). Resolution is not positional — moving a file within the scope does not change what resolves — and there is no shadowing: the pool is a union, so two files defining the **same id** in one scope is a surfaced **duplicate-id error**, never a silent pick.

4. **Failure surface.** A dangling id at an annotated position (the namespace has a pool in scope but the id is absent from it) is a **hard error** — non-zero exit, same severity as a shape violation. A **duplicate id** across bound files in scope is also a hard error (the pool is ambiguous). "No file bound to the namespace anywhere in scope" is a **distinct, softer** outcome: incremental authoring routinely references a not-yet-created sibling. (`--strict` escalation of the soft case is deferred until a need appears.)

5. **Scope of this decision.** Value-position references only. Two extensions are **deferred** to successor decisions, each pending a real consumer: (a) **cross-scope shared reference roots** — letting a shared catalog serve many instances *without* unioning the instances into one validation scope (a declared search-path entry), wanted only once a real shared-data layout exists; and (b) foreign-keyed **mapping keys** (a document schema exporting a sub-collection so another file's keys validate against it).

## Rationale

**Why "through the binding" refines COR-019 rather than contradicting it.** COR-019's rule — *resolve into the namespace owner's collection* — stays true. What changes is *where that collection lives*. Capability-side, a namespace's entries sit in the namespace-declaring schema's own YAML, so COR-019 read them there. For an adopter-data namespace the entries sit in the bound instance and the schema's own collection is deliberately empty. COR-019 predated COR-023's adopter-data layer, so it never had to distinguish the two locations. This record adds that one distinction — *locate the collection by following the binding* — and changes nothing about the token form or the capability-side behaviour, so COR-019 is cited, not amended.

**Why a position-gate annotation instead of walking all values.** Adopter data carries bracketed tokens that are *not* references — in prose fields, free-text values, or comments. A resolver that scanned every string for `[x:y]` would either error on those (false positives on legitimate, non-reference tokens) or silently ignore any namespace it doesn't recognise — which also silently ignores a *typo'd* real namespace, gutting the check. Marking the reference-bearing field removes the dilemma: the resolver looks only where the schema author declared references, so an incidental token in a free-text field is structurally out of scope, and a malformed token at a *declared* position is unambiguously an error. Key-position references already require this declaration (`x-pkit-keys-from-namespace`); the value-position annotation is its symmetric counterpart.

**Why the validated subtree is the scope.** Two positional rules were considered and rejected. *Same-directory only* forbids the layouts adopter data naturally takes — domain detail in subfolders, references between peer domains, a shared catalog — forcing everything flat or duplicated. *Nearest-ancestor* (resolve against the nearest bound file found walking up) looks more flexible but fails the same layouts: a `_shared/` catalog is a *sibling*, not an ancestor, so it is never seen; subfolder detail is *below* the citing file, never reached by an upward walk; peer domains are siblings; and "nearest wins" *fully shadows* farther files, so adding a partial local file silently breaks references to shared entries. Resolving against the whole validated subtree avoids all of these — subfolders, peer domains, and any shared file under the scope are all in the pool — and it removes positional fragility (moving a file within the scope changes nothing) and shadowing (the pool is a union; a duplicate id is surfaced, never silently picked). Isolation moves from *directory position* to *what you validate*: a per-instance run is a strict universe; a wider run deliberately unions (a shared file under the parent then serves every instance below). That matches how an author already reasons — point the validator at the data set you mean — and makes debug-switching a matter of which subtree you validate, not a schema edit. The spec stays location-agnostic; no path is baked into a schema.

**Why the soft "no bound instance" case.** Capability schema sets are authored complete, so a missing sibling there is a bug (hard error). Adopter data is authored incrementally, so a reference to a not-yet-created sibling is a normal in-progress state, not a defect — distinct and softer. A dangling id (file present, id absent) is wrong regardless, so it stays a hard error.

**Why value-position now and mapping-keys later.** The value-position case has a live consumer, and the cost of the gap is *silent referential corruption* rather than visible friction — which warrants closing it at first occurrence rather than waiting for painful recurrence. The mapping-key case has no live consumer yet; its annotation vocabulary (how a document schema exports a keyable sub-collection) should be chosen against a real second use, not guessed ([COR-007](COR-007-pattern-extraction.md)).

### Alternatives considered

- **Resolve into the namespace schema's own collection** (COR-019's capability-side behaviour, applied unchanged to adopter data). Rejected — that collection is empty by design for adopter-data schemas, so every reference would dangle.
- **Walk all string values for tokens.** Rejected — collides with non-reference bracketed tokens in prose (evidence citations), and the only escape (ignore unrecognised namespaces) silently swallows typos in real ones. The position-gate annotation is the fix.
- **Same-directory-only scope.** Rejected — forbids subfolders, peer-domain references, and shared catalogs; forces flat or duplicated data. It is the special case of scope-subtree where the instance happens to be flat.
- **Nearest-ancestor (resolve up the tree, nearest file wins).** Rejected — a sibling `_shared/` catalog is never an ancestor (unreachable), subfolder detail is below the citing file (an upward walk never reaches it), peer domains are siblings, and nearest-wins fully shadows farther files (a partial local file silently breaks shared references). Scope-subtree handles all of these and removes the positional fragility.
- **A repo-wide / global resolution pool.** Rejected — isolation should be the author's scope choice, not an unbounded pool. The scope is exactly the validated subtree; validate narrower for stricter isolation.
- **Build the mapping-key (foreign-key) case now.** Rejected as speculative — deferred until a real consumer fixes the vocabulary (COR-007).

## Implications

- **New companion annotation `x-pkit-reference-namespace`** (value-position), documented alongside `x-pkit-keys-from-namespace` (key-position) in the schemas reference. A capability whose adopter data carries references marks the citing field; the referenced namespace's schema already declares the schemas mechanism's `x-pkit-id-collection` annotation and (for binding) `binds_to:`.
- **`pkit data validate` gains a default-on reference-resolution pass**, `--shape-only` to skip (mirroring `pkit schemas validate`). It operates on parsed data values, never raw file text, so comment-borne tokens are out of scope by construction as well as by the position-gate.
- **Validation-tightening, not a migration.** A previously-green adopter tree containing a dangling reference turns red on upgrade — an adopter-observable behaviour change → a surface change that bumps the backbone version and earns a release note ([PRJ-002](../project/PRJ-002-version-bump-policy.md)). It is **not** a [COR-010](COR-010-resource-lifecycle.md) migration: the trigger list (file rename/removal in kit-owned trees, `schema_version` bump, breaking CLI signature, capability subtree restructure) does not include tightening a validator.
- **Scope is value-position references.** Foreign-keyed mapping-key validation is explicitly out of scope here and tracked for a successor decision.
- **Build sanction on acceptance.** Accepting this record sanctions: the `x-pkit-reference-namespace` annotation and its companion/`$defs` support; the binding-aware scope-subtree resolution pass in `pkit data validate` (the id pool is the union of bound files under the validated target, reusing the existing id-collection machinery for the id-check tail); duplicate-id detection; the hard/soft failure split; tests; the schemas-reference update; and a backbone version bump with a release note.
